# 通用后端数据持久化规则

本文件描述数据库、事务、迁移和缓存一致性底线。具体数据库、ORM、迁移工具、表结构和缓存组件由项目级规则定义。

## 1. 数据模型

- 数据模型变更必须区分外部契约、领域模型、持久化模型和查询视图。
- 金额、时间、状态、权限、库存、额度和计数类字段必须明确精度、时区、单位和权威来源。
- 不得凭字段名猜测业务含义；状态枚举和流转必须来自权威业务规则或接口契约。

## 2. 事务与一致性

- 多表写入、状态流转、资金、库存、订单、审批和权限变更应明确事务边界。
- 事务内避免执行不可回滚的外部副作用；必须执行时应设计补偿、重试或 outbox 机制。
- 并发一致性应依赖唯一约束、版本号、锁、状态机或幂等表等可靠机制。

## 3. 数据迁移

- schema 变更应考虑向前兼容、回填、默认值、索引构建、回滚和灰度发布。
- 大表迁移、批量回填和索引变更应评估锁表、耗时、资源占用和线上影响。
- 删除字段、表、枚举值或索引前必须确认调用方和历史数据处理策略。
- 缺少回滚、补偿、幂等或线上影响说明时，不得执行破坏性 schema、数据迁移或批量回填方案。

## 4. 查询与性能

- 查询应关注索引、分页、排序、过滤条件、N+1、慢查询和大结果集内存占用。
- 统计、导出、报表和批处理应按数据规模选择异步化、分片、游标、流式处理或离线任务。
- 不得为了临时方便绕过项目的数据访问边界和权限过滤。

## 5. 缓存

- 缓存必须定义 key、值结构、过期、刷新、穿透/击穿/雪崩防护和权限隔离。
- 写入路径必须说明缓存失效或更新策略，不能只在读取路径增加缓存。
- 账号、租户、权限、语言、地域和环境差异必须进入缓存隔离维度。
